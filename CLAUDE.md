# CLAUDE.md — tekmar-infrastructure

## Identity

You are managing the **infrastructure and deployment configuration** for
Tekmar, an AI-powered infrastructure query and analysis engine for
security and compliance teams. This repository does not contain
application code. It contains three categories of files:

1. **Contracts** — the authoritative interface specifications that govern
   how all Tekmar components communicate. These are maintained by the
   technical leadership, not generated by agents. Do not modify contract
   files unless explicitly instructed to do so.

2. **Deployment configuration** — Docker Compose files, environment
   templates, and setup scripts that define how the system runs in
   development and production environments.

3. **Documentation** — the project file structure reference and any
   operational documentation.

## Development Docker Compose

The development `docker-compose.yml` must define the following services
and their relationships. All service names and ports must match what
the application repositories expect in their default configurations.

**Services:**

| Service | Image/Build | Port | Description |
|---------|-------------|------|-------------|
| `tekmar-db` | postgres:15 | 5432 | PostgreSQL database. Shared by tekmar-api and tekmar-admin-api. |
| `tekmar-api` | Build from `../tekmar-api` | 3000 | Application API (NestJS). |
| `tekmar-pipeline` | Build from `../tekmar-pipeline` | 8100 | Pipeline Service (FastAPI). |
| `tekmar-interface` | Build from `../tekmar-interface` | 4200 | Angular dev server (or nginx for built assets). |

MCP servers (`tekmar-integrations`) are not separate Docker services in
development. They run as child processes of the Pipeline Service via
stdio transport. The Pipeline Service's Dockerfile or startup script
handles launching them.

The Admin Frontend and Admin API are not included in the development
Docker Compose for the MVP. They will be added when those repositories
are built.

**Service dependencies:**
- `tekmar-api` depends on `tekmar-db` (needs database).
- `tekmar-pipeline` depends on nothing (stateless, no DB).
- `tekmar-interface` depends on `tekmar-api` (API must be reachable).

**Volumes:**
- `tekmar-db` data is persisted via a named Docker volume so the
  database survives container restarts.
- Application source directories are mounted as volumes for hot-reload
  during development (NestJS watch mode, Angular serve, FastAPI reload).

**Networking:**
- All services share a single Docker network (`tekmar-network`).
- `tekmar-api` communicates with `tekmar-pipeline` via the Docker
  network using the service name as hostname
  (`http://tekmar-pipeline:8100`).
- `tekmar-interface` proxies API requests to `tekmar-api` (Angular
  dev server proxy config or nginx reverse proxy).

## Environment Variables Template

The `.env.example` file must document every environment variable used
by every service, grouped by service. Each variable includes a
description and a sensible default where applicable.

**tekmar-api variables** (see tekmar-api CLAUDE.md for the full list):
- `PORT`, `DATABASE_URL`, `JWT_SECRET`, `JWT_EXPIRATION`,
  `REFRESH_TOKEN_EXPIRATION_DAYS`, `PIPELINE_SERVICE_URL`,
  `CREDENTIAL_ENCRYPTION_KEY`, `FILE_STORAGE_PATH`, `CORS_ORIGIN`,
  `NODE_ENV`

**tekmar-pipeline variables** (see tekmar-pipeline CLAUDE.md for the full list):
- `PORT`, `LLM_PROVIDER`, `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`,
  `OLLAMA_BASE_URL`, `OLLAMA_MODEL`, `LLM_MAX_TOKENS`,
  `LLM_TEMPERATURE`, `MCP_SERVERS_CONFIG`,
  `EXECUTION_TIMEOUT_SECONDS`, `STEP_TIMEOUT_SECONDS`, `LOG_LEVEL`

**tekmar-db variables:**
- `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD`

The `.env.development` file contains development-safe defaults (local
database URL, a development JWT secret, etc.). It must be gitignored
to prevent accidental commit of secrets. The `.env.example` file is
committed and contains placeholder values with descriptions.

## Setup Script

The `scripts/setup.sh` script automates first-time project setup for
a new developer. It should:

1. Check prerequisites (Docker, Docker Compose, Node.js, Python, npm).
2. Clone all MVP repositories into sibling directories if they do not
   already exist.
3. Copy `.env.example` to `.env.development` if it does not exist.
4. Run `docker-compose up -d tekmar-db` to start the database.
5. Run database migrations (`cd ../tekmar-api && npx prisma migrate dev`).
6. Print instructions for starting each service.

The script should be idempotent — running it a second time should not
break anything.

## Coding Conventions

**Docker Compose** — use version 3.8+ syntax. Pin image versions (do not
use `latest` tags). Use multi-stage builds in Dockerfiles to keep images
small.

**Environment files** — `.env.example` is committed, `.env.development`
and `.env.production` are gitignored. Never commit real secrets.

**Scripts** — use bash with `set -euo pipefail` for safety. Add comments
explaining each step. Test on both macOS and Linux.

**Contract files** — do not modify unless explicitly instructed. Contract
changes require version bumps and coordinated updates across affected
repositories.
